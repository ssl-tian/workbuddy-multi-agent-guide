# 零代码小白实操指南：用 WorkBuddy 搭建你的第一个多 Agent 协同系统

> **作者的话**：我是一个精神科医生，不是程序员。这篇指南里写的每一步，都是我自己踩过坑之后验证过的——你不需要写一行代码，只需要会打字。

**GitHub**: [@ssl-tian](https://github.com/ssl-tian)

---

## 一句话理解多 Agent

想象你是主编，手下有 4 个审稿人。你把稿件发给每个人，他们从不同角度审读，最后你把意见汇总成一份修改清单。多 Agent 协同就是这个模式：你是头，Agent 是你的专家同事。

---

## 你需要准备什么

**什么都没有。** 打开 WorkBuddy 就行。不需要安装 Python、不需要配环境变量、不需要 API Key——WorkBuddy 本身就是 LLM，内置了 Agent 工具。

> 这是我在开发中踩的第一个坑：我花了两天写了一个 `llm_reviewer.py`，用来通过 HTTP 调用外部 API。后来才意识到——我在 WorkBuddy 里面啊，WorkBuddy 自己就是 LLM，为什么要绕一圈调外部 API？

## 实操六步

### 第 1 步：创建一个 Team

在 WorkBuddy 对话框里输入：

```
帮我创建一个 Team，叫"审稿团队"，描述是"多视角审稿协同"
```

WorkBuddy 会用 TeamCreate 工具帮你建好团队，创建任务列表。

**发生了什么**：WorkBuddy 在后台创建了一个团队配置文件和一个任务列表，所有 Agent 共享这个任务列表——就像共享一个 Trello 看板。

### 第 2 步：给每个 Agent 写"任务说明书"

Agent 需要知道三件事：**你是谁、看什么、怎么评**。不需要写复杂代码，用自然语言就行。

**方法学审稿人示例**：

```
你是方法学审稿人，专注评估研究设计的合理性。
评分标准（每项 0-100 分）：
- 研究设计是否匹配研究问题
- 混杂因素控制是否充分
- 统计方法选择是否正确
- 局限性讨论是否诚实

输出要求：
1. 每项评分的具体理由（引用稿件原文）
2. 列出最严重的 3 个方法学问题
3. 给出改进建议（按优先级排序）
```

就这样——把这段话保存为一个文件（比如 `reviewer_prompts.yaml`），每次派生 Agent 时喂给它。

### 第 3 步：派生 Agent 开始干活

在 WorkBuddy 里说：

```
派生 4 个 Agent 审稿人，分别加载以下 prompt：
- 方法学审稿人（文件：reviewer_prompts.yaml 中的 methodologist）
- 领域审稿人（文件：reviewer_prompts.yaml 中的 domain）
- 统计审稿人（文件：reviewer_prompts.yaml 中的 stats）
- 魔鬼代言人（文件：reviewer_prompts.yaml 中的 devils_advocate）

每个 Agent 审读附件中的稿件，返回结构化评分的 JSON 格式结果。
```

Agent 会并行运行（同时审稿，不互相等待），每个 Agent 完成后通过 SendMessage 把结果发回给你。

### 第 4 步：汇总和去重

4 个 Agent 都完成后，你会收到 4 条消息。接下来做三件事：

1. **去重**：如果两个 Agent 说了同样的问题，合并为一条
2. **识别冲突**：如果方法学说"方法合适"但统计说"方法有误"，标出来让主编裁决
3. **排序**：按严重度排序，产出最终修改清单

在 WorkBuddy 里说：

```
这是 4 份审稿意见（见上文）。请做：
1. 去重合并——语义相似的问题合并为一条
2. 识别冲突——标出两个以上审稿人意见矛盾的条目
3. 按严重度 × 共识度排序，输出最终修改清单
```

### 第 5 步：防幻觉验证（别跳！）

Agent 可能会瞎编。这是我在第 3 次审稿时发现的问题——Agent 引用了稿件里不存在的数字。

**三层检查**（可以直接让 WorkBuddy 做）：

1. **格式检查**：评分是否在 0-100 范围内？JSON 格式是否正确？
2. **事实检查**：Agent 引用的原文内容是否真的在稿件里？
3. **评分一致性**：同一稿件让 3 次独立审稿，评分方差是否过大？

在 WorkBuddy 里说：

```
对上面的审稿意见做防幻觉验证：
1. 检查所有评分是否在 0-100 范围内
2. 逐条检查 Agent 引用的原文内容是否真的在稿件里
3. 如果任何一条引用查不到原文，标记为"幻觉风险"
```

### 第 6 步：根据意见修改

把验证过的修改清单喂给 WorkBuddy，逐条修改稿件。

```
根据以下修改清单，逐条修改稿件：
（粘贴修改清单）
每改完一条，标注"已完成"。
```

## 真实踩坑记录（来自开发历史）

### 坑 1：以为需要 API Key

> **症状**：花了两天写 `llm_reviewer.py`，调 DeepSeek API，配环境变量。
> **真相**：WorkBuddy 内置的 Agent 工具直接派生审稿人就行，一行代码都不需要。
> **教训**：先确认你的环境有没有内置能力，再决定要不要引入外部依赖。

### 坑 2：Agent 不听话（输出格式混乱）

> **症状**：Prompt 写"输出 JSON"，Agent 返回了一段散文。
> **修复**：在 Prompt 里加"🔥 强制指令：你必须调用 SendMessage 工具，recipient='main'，content 为严格的 JSON 格式，不附加任何解释文字"。
> **本质**：Agent 不是不听话，是你的指令不够"强制"。

### 坑 3：Agent 瞎编数据

> **症状**：Agent 引用了"样本量 N=15,234"，但稿件里写的是 N=20,138。
> **修复**：每个审稿结果回来之后，必须跑一遍"防幻觉验证"——逐条检查引用是否真的在原文里。
> **本质**：LLM 会"合理想象"，你的验证流程就是针对这个设计的。

## 进阶路线（等你跑通上面六步后再看）

| 阶段 | 做什么 | 为什么 |
|------|--------|--------|
| 第一阶段 | 搭一个 2-Agent 的简单系统（如：写手 + 编辑） | 验证基本流程 |
| 第二阶段 | 加防幻觉验证（格式 + 内容 + 评分） | Agent 开始出错的时候你就需要了 |
| 第三阶段 | 扩展 Agent 数量（3-5 个） | 多视角才有真正的协同价值 |
| 第四阶段 | 加状态管理（用 YAML 或 JSON 文件记录进度） | 防止跨 Session 丢失状态 |

## 附：我用的文件模板

### reviewer_prompts.yaml 示例

```yaml
methodologist:
  role: "方法学审稿人"
  focus: "研究设计、因果推断方法、混杂控制"
  scoring:
    - name: "研究设计匹配度"
      range: [0, 100]
      description: "研究方法是否直接回答研究问题"
    - name: "混杂控制充分性"
      range: [0, 100]
      description: "混杂变量的识别和控制策略"
    - name: "统计方法正确性"
      range: [0, 100]
      description: "统计方法选择和应用是否正确"
    - name: "局限性诚实度"
      range: [0, 100]
      description: "是否诚实地讨论了方法局限"
  output_format: "JSON"
  anti_hallucination:
    - "所有引用必须标注稿件原文位置（段落/行号）"
    - "不确定的内容标注置信度"
    - "禁止编造稿件中不存在的数据"
```

### pipeline_state.yaml 骨架

```yaml
project:
  name: "你的项目名"
  description: "一句话描述"
  
data:
  source: "数据来源（如 NHANES 2005-2020）"
  sample_size: 0

stages:
  - name: "数据准备"
    status: "pending"
  - name: "建模分析"
    status: "pending"
  - name: "图表生成"
    status: "pending"
  - name: "论文撰写"
    status: "pending"
  - name: "审稿回复"
    status: "pending"

artifacts: []
decision_log: []
error_log: []
```

> **最后的建议**：不要一开始就追求"完美的多 Agent 系统"。从 2 个 Agent 开始，跑通一次完整的"派发→收集→汇总"循环。系统的价值不在于 Agent 数量多，而在于每个 Agent 都做了人做不了的事（不同维度的深度分析、交叉验证、防幻觉检查）。用起来，改起来，它会越来越好。
